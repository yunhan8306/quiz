# 문제 제작 & 평가 규칙 v1

> 이 문서는 문제를 **생성·검수·평가하는 기준**이다.
> 사람이 만들든 AI가 생성하든, 모든 문제는 이 규칙을 통과해야 문제 은행(`data/sections/`)에 들어간다.

---

## 1. 문제 제작 규칙 (Authoring Rules)

### 1.1 범위 규칙

- **1문제 = 1개념.** 두 개념이 섞이면 문제를 쪼갠다. 개념 간 관계는 `related`와 용어 링크로 연결한다.
- **중복 금지.** 같은 개념을 다른 표현으로 다시 묻지 않는다. 단, 같은 개념의 **다른 깊이**(정의 → 원리 → 엣지 케이스)는 별개 문제로 허용한다.
- **커버리지 배분.** 섹션당 목표 비율:
  - `core` (핵심 개념) **50%** — 모르면 면접 탈락 수준의 필수 개념
  - `deep` (심화/원리) **30%** — 내부 동작 원리, "왜?"를 묻는 문제
  - `niche` (지엽/디테일) **20%** — 세부 스펙, 엣지 케이스, 함정
- **지엽 문제의 자격.** niche는 '실무에서 실제로 마주치는' 디테일만 허용한다 (예: SharedFlow replay 버퍼 동작 O, API 버전별 상수값 암기 X). 트리비아는 출제하지 않는다.
- **난이도 정의** (`difficulty`):
  - `1` — 정의, 용도, 기본 구분 ("~란 무엇인가")
  - `2` — 비교, 동작 원리, 상황별 선택 ("A vs B", "왜 이렇게 동작하나")
  - `3` — 내부 구현, 엣지 케이스, 함정, 성능 특성

### 1.2 질문(스템) 규칙

- 스템만 읽고 **무엇을 묻는지 명확**해야 한다. 보기를 읽어야 질문이 이해되면 실격.
- **부정형**("옳지 **않은** 것은?")은 반드시 굵게 강조하고, 섹션당 20% 이하로 제한한다.
- 가능하면 **실무 시나리오형**으로 쓴다: "화면 회전 시…", "싱글톤이 ~를 참조하면…" — 맥락이 있는 문제가 기억에 남는다.
- 스템에 정답 단서를 남기지 않는다 (보기 중 하나와 같은 표현을 스템에 쓰는 것 금지).

### 1.3 보기(choices) 규칙

- 4지선다, 정답 1개.
- **오답은 '흔한 오해'에서 만든다.** 무작위 거짓말이 아니라, 실제로 헷갈리는 지점이어야 한다:
  - 반대로 뒤집기 ("IO가 아니라 Default가 스레드 풀이 크다")
  - 비슷한 개념과 바꿔치기 (remember ↔ rememberSaveable의 성질 교차)
  - 과잉 일반화 ("ViewModel은 프로세스가 죽어도 복원된다")
- **길이·문체 균일.** 정답만 길고 자세하면 안 된다. 정답이 가장 길어지는 경향을 의식적으로 깨트린다.
- "항상", "절대", "반드시", "~만" 같은 극단 표현으로 오답임을 유추할 수 있게 하지 않는다.
- 보기끼리 **상호 배타적**이어야 한다 (두 개가 동시에 참일 수 있으면 실격).
- `correctIndex` 분산: 한 섹션에서 같은 번호가 40%를 넘지 않게 한다.

### 1.4 해설(explanation) 규칙

- **해설만 읽어도 그 개념을 복습할 수 있는 분량**으로 쓴다 (모범답안 수준). 이 앱은 객관식이지만 해설이 주관식 모범답안 역할을 겸한다.
- 정답의 근거뿐 아니라 **주요 오답이 왜 틀렸는지**도 설명한다 ("②는 반대", "③은 ~라서 오답").
- 용어사전(`glossary.json`)에 있는 키워드가 해설 본문에 자연스럽게 등장하게 써서 **자동 링크**가 걸리게 한다.
- `keywords` 최소 1개, 연관 문제가 있으면 `related`를 채운다. 새 핵심 용어가 등장하면 glossary에 먼저 등록한다.

## 2. 문제 평가 루브릭 (Evaluation Rubric)

모든 문제는 4개 축을 1~5점으로 평가해 `eval` 필드에 기록한다.

### 2.1 frequency — 면접 출제 빈도 (가중치 40%)

"안드로이드 면접에서 이 주제가 얼마나 자주 나오는가"

| 점수 | 기준 |
|---|---|
| 5 | 거의 모든 면접에서 나옴 (생명주기, 코루틴 기본, 리컴포지션) |
| 3 | 절반 정도의 면접에서 나옴 (Cold/Hot Flow, Repository) |
| 1 | 드물게 나옴 (특정 라이브러리 내부, 레거시 API) |

### 2.2 centrality — 개념 중심성 (가중치 25%)

"이걸 모르면 다른 개념도 무너지는가 / 다른 개념과 얼마나 연결되는가"

| 점수 | 기준 |
|---|---|
| 5 | 여러 주제의 뿌리 (생명주기, 구조화된 동시성, 리컴포지션) |
| 3 | 한 영역 안에서 중요 (StateFlow, Hilt 스코프) |
| 1 | 고립된 지식 (특정 API 시그니처) |

### 2.3 practicality — 실무 영향도 (가중치 20%)

"모르면 실제 버그, 크래시, 성능 문제로 이어지는가"

| 점수 | 기준 |
|---|---|
| 5 | 모르면 흔한 프로덕션 버그 발생 (Context 누수, 리컴포지션 폭발, 회전 시 상태 유실) |
| 3 | 코드 품질/설계에 영향 (DI, 아키텍처 선택) |
| 1 | 알면 좋은 교양 수준 |

### 2.4 discrimination — 변별력 (가중치 15%)

"이 질문의 답변 수준이 주니어/시니어를 가르는가"

| 점수 | 기준 |
|---|---|
| 5 | 깊이가 그대로 드러남 (suspend의 CPS 변환, 컴포지션 내부) |
| 3 | 정확히 아는 사람과 대충 아는 사람이 갈림 (A vs B 비교류) |
| 1 | 외우면 누구나 답함 (용어 정의) |

### 2.5 우선순위 점수와 티어

```
priority = 0.40·frequency + 0.25·centrality + 0.20·practicality + 0.15·discrimination
```

| 티어 | 점수 | 의미 |
|---|---|---|
| **S 필수** | ≥ 4.2 | 면접 전 반드시 완벽하게. 항상 최상단 |
| **A 중요** | ≥ 3.4 | 우선 학습 대상 |
| **B 기본** | ≥ 2.6 | 기본기 확인용 |
| **C 지엽** | < 2.6 | 시간 남을 때. 항상 하단 |

- 섹션 화면에서 문제는 **priority 내림차순**으로 정렬된다 (앱이 자동 계산).
- 가중치 근거: 이 도구의 목적이 **면접 대비**이므로 출제 빈도가 최우선. 중심성이 그다음인 이유는 중심 개념 하나가 여러 질문의 답을 커버하기 때문.

## 4. 생성/검수 체크리스트

새 문제를 추가하기 전에 전부 체크:

- [ ] 1문제 1개념인가? (두 개념이면 분리)
- [ ] 같은 개념의 기존 문제와 중복되지 않는가?
- [ ] 스템만 읽고 무엇을 묻는지 명확한가?
- [ ] 오답 3개가 모두 '흔한 오해' 기반으로 그럴듯한가?
- [ ] 정답이 길이·문체로 티 나지 않는가? 극단 표현 단서가 없는가?
- [ ] 해설이 모범답안 수준이고, 주요 오답의 이유를 포함하는가?
- [ ] `eval` 4개 축을 루브릭 기준표에 맞게 채점했는가?
- [ ] `coverage`가 섹션 배분(50/30/20)을 깨뜨리지 않는가?
- [ ] `keywords`가 glossary에 실제 존재하는가? 새 용어는 glossary에 등록했는가?
- [ ] `correctIndex`가 한쪽으로 몰리지 않는가?
- [ ] `id`가 유일한가?

기계 검증(ID 중복, keywords/related 참조 무결성, correctIndex 범위)은 `python3 validate.py`로 실행한다.
