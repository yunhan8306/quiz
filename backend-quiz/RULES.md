# 문제 제작 & 평가 규칙 v1 — 백엔드 공통 지식 (기술 면접 · 직군 전환 대비)

> 이 문서는 문제를 **생성·검수·평가하는 기준**이다.
> 사람이 만들든 AI가 생성하든, 모든 문제는 이 규칙을 통과해야 문제 은행(`data/sections/`)에 들어간다.
> 이 퀴즈는 **언어(Java/Kotlin/Python)·프레임워크(Spring/Django/Express) 무관 공통 지식**만 다룬다.
> 개념 자체가 범용이면 포함(예: ORM의 N+1 문제), 특정 스택 고유 문법·API는 제외(예: JPA 어노테이션, Express 미들웨어 시그니처).

---

## 1. 문제 제작 규칙 (Authoring Rules)

### 1.1 범위 규칙

- **1문제 = 1개념.** 두 개념이 섞이면 문제를 쪼갠다. 개념 간 관계는 `related`와 용어 링크로 연결한다.
- **중복 금지.** 같은 개념을 다른 표현으로 다시 묻지 않는다. 단, 같은 개념의 **다른 깊이**(정의 → 원리 → 응용/예외)는 별개 문제로 허용한다.
- **커버리지 배분.** 섹션당 목표 비율:
  - `core` (핵심 개념) **50%** — 모르면 백엔드 개발자 면접에서 탈락 수준의 필수 개념
  - `deep` (심화/원리) **30%** — 동작 원리·배경, "왜?"를 묻는 문제
  - `niche` (지엽/디테일) **20%** — 세부 사항, 예외 케이스, 함정
- **지엽 문제의 자격.** niche는 '백엔드 실무에서 실제로 마주치는' 디테일만 허용한다. 단순 트리비아(연도·수치 암기 등 이해와 무관한 것)는 출제하지 않는다.
  - 허용 예: "인덱스가 걸린 컬럼인데도 옵티마이저가 풀 테이블 스캔을 선택하는 경우" — 실무 쿼리 튜닝에서 실제로 마주친다.
  - 금지 예: "HTTP/2를 정의한 RFC 문서 번호는?" — 이해와 무관한 암기.
- **난이도 정의** (`difficulty`): 목표 분포 **1: 20% / 2: 55% / 3: 25%** (중급 중심 — 경력 개발자의 직군 전환 면접 기준).
  - `1` — 정의, 용도, 기본 구분
  - `2` — 비교, 원리, 상황별 판단
  - `3` — 내부 구조/심화 원리, 예외 케이스, 함정

### 1.2 질문(스템) 규칙

- 스템만 읽고 **무엇을 묻는지 명확**해야 한다. 보기를 읽어야 질문이 이해되면 실격.
- **부정형**("옳지 **않은** 것은?")은 반드시 굵게 강조하고, 섹션당 20% 이하로 제한한다.
- 가능하면 **실전 시나리오형**으로 쓴다 — 맥락이 있는 문제가 기억에 남는다.
  - 예: "트래픽 급증 후 API 서버가 DB 커넥션을 얻지 못해 응답이 밀리기 시작했다. 가장 먼저 의심할 지점은?"
- 스템에 정답 단서를 남기지 않는다 (보기 중 하나와 같은 표현을 스템에 쓰는 것 금지).

### 1.3 보기(choices) 규칙

- 4지선다, 정답 1개.
- **오답은 '흔한 오해'에서 만든다.** 무작위 거짓말이 아니라, 실제로 헷갈리는 지점이어야 한다:
  - 반대로 뒤집기
  - 비슷한 개념과 바꿔치기
  - 과잉 일반화
  - 이 분야의 흔한 오해 예:
    - "TCP는 메시지 경계를 보존한다" (스트림 프로토콜인데 메시지 단위로 착각)
    - "JWT는 암호화되어 있어서 페이로드 내용을 볼 수 없다" (서명과 암호화의 혼동)
    - "인덱스는 많이 걸수록 조회가 빨라지니 좋다" (쓰기 비용·옵티마이저 혼란 무시)
- **길이·문체 균일.** 정답만 길고 자세하면 안 된다 (정답이 오답 평균 길이의 1.8배 미만).
- "항상", "절대", "반드시", "~만" 같은 극단 표현으로 오답임을 유추할 수 있게 하지 않는다.
- 보기끼리 **상호 배타적**이어야 한다 (두 개가 동시에 참일 수 있으면 실격).
- `correctIndex` 분산: 한 섹션에서 같은 번호가 40%를 넘지 않게 한다.

### 1.4 해설(explanation) 규칙

- **해설만 읽어도 그 개념을 복습할 수 있는 분량**으로 쓴다 (모범답안 수준). 객관식이지만 해설이 서술형 모범답안 역할을 겸한다 — 면접에서 그대로 말할 수 있는 수준을 목표로.
- 정답의 근거뿐 아니라 **주요 오답이 왜 틀렸는지**도 설명한다.
- 용어사전(`glossary.json`)에 있는 키워드가 해설 본문에 자연스럽게 등장하게 써서 **자동 링크**가 걸리게 한다.
- `keywords` 최소 1개, 연관 문제가 있으면 `related`를 채운다. 새 핵심 용어가 등장하면 glossary에 먼저 등록한다.

## 2. 문제 평가 루브릭 (Evaluation Rubric)

모든 문제는 4개 축을 1~5점으로 평가해 `eval` 필드에 기록한다.

### 2.1 frequency — 출제 빈도 (가중치 40%)

"백엔드 개발자 채용 면접에서 이 주제가 얼마나 자주 나오는가"

| 점수 | 기준 |
|---|---|
| 5 | 거의 매번 나옴 — TCP 3-way handshake, HTTP 메서드/상태코드, 트랜잭션 격리수준, 인덱스 동작 원리, 세션 vs JWT |
| 3 | 절반 정도 나옴 — 일관된 해싱(consistent hashing), CAP 정리, 캐시 무효화 전략, 데드락 조건 |
| 1 | 드물게 나옴 — BGP 경로 선택 세부 규칙, 특정 파일시스템 저널링 구현 |

### 2.2 centrality — 개념 중심성 (가중치 25%)

"이걸 모르면 다른 개념도 무너지는가 / 다른 개념과 얼마나 연결되는가"

| 점수 | 기준 |
|---|---|
| 5 | 여러 주제의 뿌리 — 트랜잭션(격리수준·락·복제·분산 트랜잭션의 전제), HTTP 요청-응답 모델(API·캐시·인증의 전제) |
| 3 | 한 영역 안에서 중요 — 커넥션 풀 동작, 캐시 축출(eviction) 정책 |
| 1 | 고립된 지식 — 특정 도구의 설정 파일 형식 |

### 2.3 practicality — 실무 장애 영향도 (가중치 20%)

"모르면 실제 서비스에서 장애·데이터 정합성 사고·성능 저하로 이어지는가"

| 점수 | 기준 |
|---|---|
| 5 | 모르면 장애/정합성 사고 직결 — 격리수준 오해로 인한 중복 결제, 커넥션 풀 고갈, 캐시 스탬피드 |
| 3 | 성능 저하·비효율 설계로 이어짐 — N+1 쿼리, 잘못된 인덱스 설계, 불필요한 동기 호출 |
| 1 | 알아도 실무 차이가 거의 없음 — 프로토콜 역사, 이론적 극단 케이스 |

### 2.4 discrimination — 변별력 (가중치 15%)

"이 질문의 답변 수준이 초심자/숙련자를 가르는가"

| 점수 | 기준 |
|---|---|
| 5 | 이해 깊이가 그대로 드러남 — "인덱스가 있는데 왜 풀 스캔이 됐는가", "복제 지연 중 자기 글이 안 보이는 문제를 어떻게 푸는가" |
| 3 | 정확히 아는 사람과 대충 아는 사람이 갈림 |
| 1 | 외우면 누구나 답함 (용어 정의) |

### 2.5 우선순위 점수와 티어

```
priority = 0.40·frequency + 0.25·centrality + 0.20·practicality + 0.15·discrimination
```

| 티어 | 점수 | 의미 |
|---|---|---|
| **S 필수** | ≥ 4.2 | 반드시 완벽하게. 항상 최상단 |
| **A 중요** | ≥ 3.4 | 우선 학습 대상 |
| **B 기본** | ≥ 2.6 | 기본기 확인용 |
| **C 지엽** | < 2.6 | 시간 남을 때. 항상 하단 |

- 섹션 화면에서 문제는 **priority 내림차순**으로 정렬된다 (앱이 자동 계산).
- ⚠️ 가중치를 기본값(40/25/20/15)에서 바꾸면 `app.js`의 `EVAL_WEIGHTS`와 `validate.py`의 `WEIGHTS`도 동일하게 수정할 것.

## 4. 생성/검수 체크리스트

새 문제를 추가하기 전에 전부 체크:

- [ ] 1문제 1개념인가? (두 개념이면 분리)
- [ ] 같은 개념의 기존 문제와 중복되지 않는가?
- [ ] 언어/프레임워크 고유 내용이 아닌가? (범용 개념으로 일반화됐는가)
- [ ] 스템만 읽고 무엇을 묻는지 명확한가?
- [ ] 오답 3개가 모두 '흔한 오해' 기반으로 그럴듯한가?
- [ ] 정답이 길이·문체로 티 나지 않는가? 극단 표현 단서가 없는가?
- [ ] 해설이 모범답안 수준이고, 주요 오답의 이유를 포함하는가?
- [ ] `eval` 4개 축을 루브릭 기준표에 맞게 채점했는가? (점수 인플레이션 금지)
- [ ] `coverage`가 섹션 배분(50/30/20)을 깨뜨리지 않는가?
- [ ] `keywords`가 glossary에 실제 존재하는가? 새 용어는 glossary에 등록했는가?
- [ ] `correctIndex`가 한쪽으로 몰리지 않는가?
- [ ] `id`가 유일한가?

기계 검증(ID 중복, keywords/related 참조 무결성, correctIndex 범위·분포, 정답 길이 티, 부정형 비율)은 `python3 validate.py`로 실행한다.
