# 문제 제작 & 평가 규칙 v1 — Product Owner / Product Management (직무 학습)

> 이 문서는 문제를 **생성·검수·평가하는 기준**이다.
> 사람이 만들든 AI가 생성하든, 모든 문제는 이 규칙을 통과해야 문제 은행(`data/sections/`)에 들어간다.
> 대상: 입문~중급 PO/PM 학습자. 목적: 직무 학습·역량 강화 (시험 대비 아님 — 실무 판단력 중심).

---

## 1. 문제 제작 규칙 (Authoring Rules)

### 1.1 범위 규칙

- **1문제 = 1개념.** 두 개념이 섞이면 문제를 쪼갠다. 개념 간 관계는 `related`와 용어 링크로 연결한다.
- **중복 금지.** 같은 개념을 다른 표현으로 다시 묻지 않는다. 단, 같은 개념의 **다른 깊이**(정의 → 원리 → 응용/예외)는 별개 문제로 허용한다.
- **커버리지 배분.** 섹션당 목표 비율:
  - `core` (핵심 개념) **50%** — 모르면 PO 실무 대화에 끼지 못하는 수준의 필수 개념
  - `deep` (심화/원리) **30%** — 동작 원리·배경, "왜 이 방식인가?"를 묻는 문제
  - `niche` (지엽/디테일) **20%** — 세부 사항, 예외 케이스, 함정
- **지엽 문제의 자격.** niche는 'PO 실무에서 실제로 마주치는' 디테일만 허용한다. 단순 트리비아(연도·수치 암기 등 이해와 무관한 것)는 출제하지 않는다.
  - 허용 예: "스프린트 중 요구사항 변경 요청이 왔을 때 스프린트 목표를 해치지 않는 처리 방식" — 실무에서 매주 마주치는 경계 케이스
  - 금지 예: "스크럼 가이드 2020년판이 이전 판보다 줄어든 페이지 수는?" — 이해와 무관한 트리비아
- **난이도 정의** (`difficulty`): 대상이 입문~중급이므로 1:2:3 ≈ 30:50:20 분포를 권장.
  - `1` — 정의, 용도, 기본 구분 (예: MVP의 정의, 스프린트 리뷰와 회고의 구분)
  - `2` — 비교, 원리, 상황별 판단 (예: RICE와 Kano 중 어느 상황에 무엇을 쓰나)
  - `3` — 심화 원리, 예외 케이스, 함정 (예: 허영 지표가 실험 설계를 오염시키는 경로)

### 1.2 질문(스템) 규칙

- 스템만 읽고 **무엇을 묻는지 명확**해야 한다. 보기를 읽어야 질문이 이해되면 실격.
- **부정형**("옳지 **않은** 것은?")은 반드시 굵게 강조하고, 섹션당 20% 이하로 제한한다.
- 가능하면 **실전 시나리오형**으로 쓴다 — 맥락이 있는 문제가 기억에 남는다.
  - 시나리오 예: "출시 3개월 차 구독 서비스의 가입자는 늘지만 첫 주 이탈률이 60%다. PO가 가장 먼저 확인해야 할 것은?"
- 스템에 정답 단서를 남기지 않는다 (보기 중 하나와 같은 표현을 스템에 쓰는 것 금지).

### 1.3 보기(choices) 규칙

- 4지선다, 정답 1개.
- **오답은 '흔한 오해'에서 만든다.** 무작위 거짓말이 아니라, 실제로 헷갈리는 지점이어야 한다:
  - 반대로 뒤집기
  - 비슷한 개념과 바꿔치기
  - 과잉 일반화
  - 이 분야의 흔한 오해 예:
    - "MVP는 완성도가 낮은 첫 출시 버전이다" (→ 실제로는 학습을 위한 최소 실험 단위)
    - "PO는 이해관계자의 요구사항을 받아 백로그에 옮기는 역할이다" (→ 요구사항 전달자가 아니라 가치 판단의 주체)
    - "A/B 테스트에서 지표가 오르면 무조건 배포한다" (→ 통계적 유의성·가드레일 지표·장기 효과 검토가 먼저)
- **길이·문체 균일.** 정답만 길고 자세하면 안 된다 (정답이 오답 평균 길이의 1.8배 미만).
- "항상", "절대", "반드시", "~만" 같은 극단 표현으로 오답임을 유추할 수 있게 하지 않는다.
- 보기끼리 **상호 배타적**이어야 한다 (두 개가 동시에 참일 수 있으면 실격).
- `correctIndex` 분산: 한 섹션에서 같은 번호가 40%를 넘지 않게 한다.

### 1.4 해설(explanation) 규칙

- **해설만 읽어도 그 개념을 복습할 수 있는 분량**으로 쓴다 (모범답안 수준). 객관식이지만 해설이 서술형 모범답안 역할을 겸한다.
- 정답의 근거뿐 아니라 **주요 오답이 왜 틀렸는지**도 설명한다.
- 용어사전(`glossary.json`)에 있는 키워드가 해설 본문에 자연스럽게 등장하게 써서 **자동 링크**가 걸리게 한다.
- `keywords` 최소 1개, 연관 문제가 있으면 `related`를 채운다. 새 핵심 용어가 등장하면 glossary에 먼저 등록한다.
- 방법론 이름이 책·저자에 따라 다르게 불리면 해설에서 병기한다 (예: 기회 솔루션 트리 = Opportunity Solution Tree).

## 2. 문제 평가 루브릭 (Evaluation Rubric)

모든 문제는 4개 축을 1~5점으로 평가해 `eval` 필드에 기록한다.

### 2.1 frequency — 실무 빈도 (가중치 40%)

"PO 실무(일상 업무·의사결정)에서 이 주제를 얼마나 자주 마주치는가"

| 점수 | 기준 |
|---|---|
| 5 | 거의 매일/매 스프린트 마주침 (예: 백로그 우선순위 판단, 유저 스토리 작성, 스프린트 이벤트) |
| 3 | 분기·프로젝트 단위로 마주침 (예: 로드맵 수립, A/B 테스트 설계, 신규 디스커버리 착수) |
| 1 | 특수 상황에서만 마주침 (예: 피벗 결정, 제품 단종 판단) |

### 2.2 centrality — 개념 중심성 (가중치 25%)

"이걸 모르면 다른 개념도 무너지는가 / 다른 개념과 얼마나 연결되는가"

| 점수 | 기준 |
|---|---|
| 5 | 여러 주제의 뿌리 (예: 아웃컴 vs 아웃풋 구분 — 전략·지표·우선순위 전체를 관통) |
| 3 | 한 영역 안에서 중요 (예: 코호트 분석 — 지표 영역의 핵심이지만 다른 영역엔 간접 영향) |
| 1 | 고립된 지식 (예: 특정 프레임워크의 세부 명칭) |

### 2.3 practicality — 실무 영향도 (가중치 20%)

"모르면 실제 제품 의사결정 실수·팀 낭비로 이어지는가"

| 점수 | 기준 |
|---|---|
| 5 | 모르면 분기 단위 낭비·잘못된 제품 방향으로 직결 (예: 가설 검증 없이 빌드 → 빌드 트랩) |
| 3 | 모르면 스프린트 단위 비효율·팀 마찰 (예: 인수 기준 없는 스토리로 개발 재작업) |
| 1 | 몰라도 당장 실무 손해는 없음 (교양·배경 지식) |

### 2.4 discrimination — 변별력 (가중치 15%)

"이 질문의 답변 수준이 초심자/숙련자를 가르는가"

| 점수 | 기준 |
|---|---|
| 5 | 이해 깊이가 그대로 드러남 (예: '이 상황에서 어떤 지표가 허영 지표인가'를 판단하는 문제) |
| 3 | 정확히 아는 사람과 대충 아는 사람이 갈림 |
| 1 | 외우면 누구나 답함 (용어 정의) |

### 2.5 우선순위 점수와 티어

```
priority = 0.4·frequency + 0.25·centrality + 0.2·practicality + 0.15·discrimination
```

| 티어 | 점수 | 의미 |
|---|---|---|
| **S 필수** | ≥ 4.2 | 반드시 완벽하게. 항상 최상단 |
| **A 중요** | ≥ 3.4 | 우선 학습 대상 |
| **B 기본** | ≥ 2.6 | 기본기 확인용 |
| **C 지엽** | < 2.6 | 시간 남을 때. 항상 하단 |

- 섹션 화면에서 문제는 **priority 내림차순**으로 정렬된다 (앱이 자동 계산).
- ⚠️ 가중치를 기본값(40/25/20/15)에서 바꾸면 `app.js`의 `EVAL_WEIGHTS`와 `validate.py`의 `WEIGHTS`도 동일하게 수정할 것.

## 4. 생성/검수 체크리스트

새 문제를 추가하기 전에 전부 체크:

- [ ] 1문제 1개념인가? (두 개념이면 분리)
- [ ] 같은 개념의 기존 문제와 중복되지 않는가?
- [ ] 스템만 읽고 무엇을 묻는지 명확한가?
- [ ] 오답 3개가 모두 '흔한 오해' 기반으로 그럴듯한가?
- [ ] 정답이 길이·문체로 티 나지 않는가? 극단 표현 단서가 없는가?
- [ ] 해설이 모범답안 수준이고, 주요 오답의 이유를 포함하는가?
- [ ] `eval` 4개 축을 루브릭 기준표에 맞게 채점했는가? (점수 인플레이션 금지)
- [ ] `coverage`가 섹션 배분(50/30/20)을 깨뜨리지 않는가?
- [ ] `keywords`가 glossary에 실제 존재하는가? 새 용어는 glossary에 등록했는가?
- [ ] `correctIndex`가 한쪽으로 몰리지 않는가?
- [ ] `id`가 유일한가?

기계 검증(ID 중복, keywords/related 참조 무결성, correctIndex 범위·분포, 정답 길이 티, 부정형 비율)은 `python3 validate.py`로 실행한다.

## 5. PO 분야 특수 규칙

- **방법론 간 충돌 주의.** 저자·프레임워크마다 권장이 다른 주제(예: 로드맵 형식, 추정 방식)는 "무엇이 옳은가"가 아니라 "이 방법론에서는 어떻게 하는가" 또는 "이 상황에는 무엇이 적합한가"로 출제한다. 특정 학파의 주장을 보편 진리처럼 출제하지 않는다.
- **스크럼 용어는 스크럼 가이드 2020 기준.** (예: '스프린트 목표'는 스프린트 백로그의 확약(commitment), 개발팀 → 개발자(Developers) 명칭 변경 반영)
- **한국 실무 맥락 허용.** 국내 IT 조직의 PO/PM 역할 구분처럼 한국 시장 특유의 맥락도 출제 가능하나, 회사별로 다른 부분은 "일반적으로"가 아니라 맥락을 명시해 출제한다.
