# 문제 제작 & 평가 규칙 v1 — 마케팅 (IT 기업 실무 중심, 개인 학습)

> 이 문서는 문제를 **생성·검수·평가하는 기준**이다.
> 사람이 만들든 AI가 생성하든, 모든 문제는 이 규칙을 통과해야 문제 은행(`data/sections/`)에 들어간다.
> 대상: IT 기업에서 일하는 비마케터(개발자 등)가 마케팅 도메인을 이해하기 위한 개인 학습. 수준: 입문~중급.

---

## 1. 문제 제작 규칙 (Authoring Rules)

### 1.1 범위 규칙

- **1문제 = 1개념.** 두 개념이 섞이면 문제를 쪼갠다. 개념 간 관계는 `related`와 용어 링크로 연결한다.
- **중복 금지.** 같은 개념을 다른 표현으로 다시 묻지 않는다. 단, 같은 개념의 **다른 깊이**(정의 → 원리 → 응용/예외)는 별개 문제로 허용한다.
- **커버리지 배분.** 섹션당 목표 비율:
  - `core` (핵심 개념) **50%** — 모르면 마케팅 관련 대화·문서를 따라갈 수 없는 필수 개념 (예: 퍼널, 리텐션, ROAS, 포지셔닝)
  - `deep` (심화/원리) **30%** — 동작 원리·배경, "왜?"를 묻는 문제 (예: 왜 마지막 클릭 어트리뷰션이 상위 퍼널 채널을 과소평가하는가)
  - `niche` (지엽/디테일) **20%** — 세부 사항, 예외 케이스, 함정
- **지엽 문제의 자격.** niche는 'IT 기업 실무에서 실제로 마주치는' 디테일만 허용한다.
  - 허용 예: iOS ATT/개인정보 보호 강화가 광고 어트리뷰션에 미친 영향, A/B 테스트에서 신뢰수준을 채우기 전에 결과를 훔쳐보는(peeking) 문제
  - 금지 예: "4P 개념을 제안한 학자의 이름과 발표 연도", 특정 광고 플랫폼의 일시적 UI 메뉴 위치 — 이해와 무관한 트리비아
- **버전/시점 기준.** 급변하는 세부 정책(특정 플랫폼의 요금·베타 기능)은 출제하지 않는다. 개념·구조 수준에서 안정된 지식만 다룬다.
- **난이도 정의** (`difficulty`):
  - `1` — 정의, 용도, 기본 구분
  - `2` — 비교, 원리, 상황별 판단
  - `3` — 내부 구조/심화 원리, 예외 케이스, 함정

### 1.2 질문(스템) 규칙

- 스템만 읽고 **무엇을 묻는지 명확**해야 한다. 보기를 읽어야 질문이 이해되면 실격.
- **부정형**("옳지 **않은** 것은?")은 반드시 굵게 강조하고, 섹션당 20% 이하로 제한한다.
- 가능하면 **실전 시나리오형**으로 쓴다 — 예: "신규 가입은 늘고 있는데 4주 차 리텐션이 계속 하락하는 앱 서비스가 있다. 가장 먼저 점검해야 할 것은?"
- 스템에 정답 단서를 남기지 않는다 (보기 중 하나와 같은 표현을 스템에 쓰는 것 금지).

### 1.3 보기(choices) 규칙

- 4지선다, 정답 1개.
- **오답은 '흔한 오해'에서 만든다.** 무작위 거짓말이 아니라, 실제로 헷갈리는 지점이어야 한다:
  - 반대로 뒤집기
  - 비슷한 개념과 바꿔치기
  - 과잉 일반화
  - 이 분야의 흔한 오해 예시:
    - "ROAS가 높은 채널에 예산을 더 쓰면 항상 총이익이 늘어난다" (한계 효율 체감·증분성(incrementality) 무시)
    - "상관관계가 확인되면 그 지표를 개선하는 캠페인은 성과를 낸다" (상관≠인과)
    - "CAC와 CPA는 같은 지표다" (고객 획득 전체 비용 vs 특정 행동당 비용 혼동)
    - "리텐션이 나쁘면 푸시 알림을 늘리면 된다" (제품 가치 문제를 채널 물량으로 덮으려는 오해)
- **길이·문체 균일.** 정답만 길고 자세하면 안 된다 (정답이 오답 평균 길이의 1.8배 미만).
- "항상", "절대", "반드시", "~만" 같은 극단 표현으로 오답임을 유추할 수 있게 하지 않는다.
- 보기끼리 **상호 배타적**이어야 한다 (두 개가 동시에 참일 수 있으면 실격).
- `correctIndex` 분산: 한 섹션에서 같은 번호가 40%를 넘지 않게 한다.

### 1.4 해설(explanation) 규칙

- **해설만 읽어도 그 개념을 복습할 수 있는 분량**으로 쓴다 (모범답안 수준). 객관식이지만 해설이 서술형 모범답안 역할을 겸한다.
- 정답의 근거뿐 아니라 **주요 오답이 왜 틀렸는지**도 설명한다.
- 용어사전(`glossary.json`)에 있는 키워드가 해설 본문에 자연스럽게 등장하게 써서 **자동 링크**가 걸리게 한다.
- `keywords` 최소 1개, 연관 문제가 있으면 `related`를 채운다. 새 핵심 용어가 등장하면 glossary에 먼저 등록한다.
- 영문 용어는 한국어 표기 뒤 괄호로 원어를 병기한다 — 예: "어트리뷰션(attribution)". 실무에서 영문 그대로 쓰는 용어(ROAS, LTV, GTM)는 영문 표기를 유지한다.

## 2. 문제 평가 루브릭 (Evaluation Rubric)

모든 문제는 4개 축을 1~5점으로 평가해 `eval` 필드에 기록한다.

### 2.1 frequency — 등장 빈도 (가중치 40%)

"IT 기업의 실무 대화·문서·마케팅 학습 자료에서 이 주제가 얼마나 자주 등장하는가"

| 점수 | 기준 |
|---|---|
| 5 | 거의 매번 등장 — 퍼널/AARRR, 리텐션, CAC·LTV, ROAS, A/B 테스트, 포지셔닝 |
| 3 | 절반 정도 등장 — 어트리뷰션 모델 비교, 코호트 분석, NPS, GTM 전략 |
| 1 | 드물게 등장 — 특정 입찰 방식의 세부 동작, 미디어 믹스 모델링(MMM) 세부 기법 |

### 2.2 centrality — 개념 중심성 (가중치 25%)

"이걸 모르면 다른 개념도 무너지는가 / 다른 개념과 얼마나 연결되는가"

| 점수 | 기준 |
|---|---|
| 5 | 여러 주제의 뿌리 — 퍼널 모델(획득·활성화·리텐션·수익·추천 전체의 틀), 세그멘테이션(타겟팅·CRM·개인화의 기반) |
| 3 | 한 영역 안에서 중요 — 어트리뷰션 윈도우(퍼포먼스 영역 내 핵심), 온보딩 활성화(그로스 영역 내 핵심) |
| 1 | 고립된 지식 — 특정 광고 소재 규격, 개별 툴의 기능명 |

### 2.3 practicality — 실무 영향도 (가중치 20%)

"모르면 IT 기업 실무(협업·의사결정·지표 해석)에서 실제 오판으로 이어지는가"

| 점수 | 기준 |
|---|---|
| 5 | 모르면 지표를 잘못 해석하거나 예산·우선순위를 오판 — 상관≠인과, ROAS의 함정, 통계적 유의성 오해 |
| 3 | 협업 대화에서 어긋나거나 요구사항을 오해 — 어트리뷰션 용어, 캠페인 구조, 세그먼트 정의 |
| 1 | 몰라도 실무 지장 거의 없음 — 마케팅사(史)적 배경, 이론가 이름 |

### 2.4 discrimination — 변별력 (가중치 15%)

"이 질문의 답변 수준이 초심자/숙련자를 가르는가"

| 점수 | 기준 |
|---|---|
| 5 | 이해 깊이가 그대로 드러남 — "리텐션 커브가 평평해지는 지점의 의미", "증분성(incrementality)과 ROAS의 차이" |
| 3 | 정확히 아는 사람과 대충 아는 사람이 갈림 |
| 1 | 외우면 누구나 답함 (용어 정의) |

### 2.5 우선순위 점수와 티어

```
priority = 0.4·frequency + 0.25·centrality + 0.2·practicality + 0.15·discrimination
```

| 티어 | 점수 | 의미 |
|---|---|---|
| **S 필수** | ≥ 4.2 | 반드시 완벽하게. 항상 최상단 |
| **A 중요** | ≥ 3.4 | 우선 학습 대상 |
| **B 기본** | ≥ 2.6 | 기본기 확인용 |
| **C 지엽** | < 2.6 | 시간 남을 때. 항상 하단 |

- 섹션 화면에서 문제는 **priority 내림차순**으로 정렬된다 (앱이 자동 계산).
- ⚠️ 가중치를 기본값(40/25/20/15)에서 바꾸면 `app.js`의 `EVAL_WEIGHTS`와 `validate.py`의 `WEIGHTS`도 동일하게 수정할 것.

## 3. 스키마 (평가 필드 포함)

```json
{
  "id": "gr-001",
  "type": "choice",
  "difficulty": 2,
  "coverage": "core",
  "eval": { "frequency": 5, "centrality": 4, "practicality": 5, "discrimination": 3 },
  "question": "…",
  "choices": ["…", "…", "…", "…"],
  "correctIndex": 2,
  "explanation": "…",
  "keywords": ["…"],
  "related": ["…"]
}
```

섹션 id 접두사: `st`(strategy-consumer), `gr`(funnel-growth), `pf`(performance-marketing), `pmm`(product-marketing), `crm`(crm-retention)

## 4. 생성/검수 체크리스트

새 문제를 추가하기 전에 전부 체크:

- [ ] 1문제 1개념인가? (두 개념이면 분리)
- [ ] 같은 개념의 기존 문제와 중복되지 않는가?
- [ ] 스템만 읽고 무엇을 묻는지 명확한가?
- [ ] 오답 3개가 모두 '흔한 오해' 기반으로 그럴듯한가?
- [ ] 정답이 길이·문체로 티 나지 않는가? 극단 표현 단서가 없는가?
- [ ] 해설이 모범답안 수준이고, 주요 오답의 이유를 포함하는가?
- [ ] `eval` 4개 축을 루브릭 기준표에 맞게 채점했는가? (점수 인플레이션 금지)
- [ ] `coverage`가 섹션 배분(50/30/20)을 깨뜨리지 않는가?
- [ ] `keywords`가 glossary에 실제 존재하는가? 새 용어는 glossary에 등록했는가?
- [ ] `correctIndex`가 한쪽으로 몰리지 않는가?
- [ ] `id`가 유일한가?

기계 검증(ID 중복, keywords/related 참조 무결성, correctIndex 범위·분포, 정답 길이 티, 부정형 비율)은 `python3 validate.py`로 실행한다.
