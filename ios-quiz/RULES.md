# 문제 제작 & 평가 규칙 v1 — iOS 개발 (기술 면접 대비)

> 이 문서는 문제를 **생성·검수·평가하는 기준**이다.
> 사람이 만들든 AI가 생성하든, 모든 문제는 이 규칙을 통과해야 문제 은행(`data/sections/`)에 들어간다.
> 버전 기준: **Swift 5.9+ / Swift 6 (strict concurrency), iOS 17+ SwiftUI** — deprecated API(예: `ObservableObject` 이전의 수동 KVO 바인딩)는 "왜 바뀌었나" 맥락으로만 출제.

---

## 1. 문제 제작 규칙 (Authoring Rules)

### 1.1 범위 규칙

- **1문제 = 1개념.** 두 개념이 섞이면 문제를 쪼갠다. 개념 간 관계는 `related`와 용어 링크로 연결한다.
- **중복 금지.** 같은 개념을 다른 표현으로 다시 묻지 않는다. 단, 같은 개념의 **다른 깊이**(정의 → 원리 → 엣지 케이스)는 별개 문제로 허용한다.
- **커버리지 배분.** 섹션당 목표 비율:
  - `core` (핵심 개념) **50%** — 모르면 iOS 면접 탈락 수준의 필수 개념
  - `deep` (심화/원리) **30%** — 내부 동작 원리, "왜?"를 묻는 문제
  - `niche` (지엽/디테일) **20%** — 세부 스펙, 엣지 케이스, 함정
- **지엽 문제의 자격.** niche는 '실무에서 실제로 마주치는' 디테일만 허용한다 (예: `@State` 초기값이 뷰 재생성 시 무시되는 동작 O, iOS 버전별 SDK 상수값 암기 X). 트리비아는 출제하지 않는다.
- **난이도 정의** (`difficulty`):
  - `1` — 정의, 용도, 기본 구분 ("~란 무엇인가")
  - `2` — 비교, 동작 원리, 상황별 선택 ("struct vs class", "왜 이렇게 동작하나")
  - `3` — 내부 구현, 엣지 케이스, 함정, 성능 특성

### 1.2 질문(스템) 규칙

- 스템만 읽고 **무엇을 묻는지 명확**해야 한다. 보기를 읽어야 질문이 이해되면 실격.
- **부정형**("옳지 **않은** 것은?")은 반드시 굵게 강조하고, 섹션당 20% 이하로 제한한다.
- 가능하면 **실무 시나리오형**으로 쓴다: "클로저가 self를 강하게 캡처하면…", "리스트 스크롤 중 뷰가 재생성되면 @State는…" — 맥락이 있는 문제가 기억에 남는다.
- 스템에 정답 단서를 남기지 않는다 (보기 중 하나와 같은 표현을 스템에 쓰는 것 금지).

### 1.3 보기(choices) 규칙

- 4지선다, 정답 1개.
- **오답은 '흔한 오해'에서 만든다.** 무작위 거짓말이 아니라, 실제로 헷갈리는 지점이어야 한다:
  - 반대로 뒤집기 ("weak가 아니라 unowned가 nil이 되면 안전하다")
  - 비슷한 개념과 바꿔치기 (`@StateObject` ↔ `@ObservedObject`의 생명주기 교차, `Task` ↔ `Task.detached`)
  - 과잉 일반화 ("struct는 항상 스택에 저장된다", "actor 안의 코드는 전부 순차 실행이므로 await 지점에서도 상태가 보존된다")
  - Android/타 플랫폼 개념의 오적용 ("SwiftUI 뷰는 Compose처럼 recomposition 스코프 단위로만 다시 그린다")
- **길이·문체 균일.** 정답만 길고 자세하면 안 된다 (정답이 오답 평균 길이의 1.8배 미만).
- "항상", "절대", "반드시", "~만" 같은 극단 표현으로 오답임을 유추할 수 있게 하지 않는다.
- 보기끼리 **상호 배타적**이어야 한다 (두 개가 동시에 참일 수 있으면 실격).
- `correctIndex` 분산: 한 섹션에서 같은 번호가 40%를 넘지 않게 한다.

### 1.4 해설(explanation) 규칙

- **해설만 읽어도 그 개념을 복습할 수 있는 분량**으로 쓴다 (모범답안 수준). 객관식이지만 해설이 서술형 모범답안 역할을 겸한다.
- 정답의 근거뿐 아니라 **주요 오답이 왜 틀렸는지**도 설명한다 ("②는 반대", "③은 ~라서 오답").
- 용어사전(`glossary.json`)에 있는 키워드가 해설 본문에 자연스럽게 등장하게 써서 **자동 링크**가 걸리게 한다.
- `keywords` 최소 1개, 연관 문제가 있으면 `related`를 채운다. 새 핵심 용어가 등장하면 glossary에 먼저 등록한다.

## 2. 문제 평가 루브릭 (Evaluation Rubric)

모든 문제는 4개 축을 1~5점으로 평가해 `eval` 필드에 기록한다.

### 2.1 frequency — 면접 출제 빈도 (가중치 40%)

"iOS 개발자 기술 면접에서 이 주제가 얼마나 자주 나오는가"

| 점수 | 기준 |
|---|---|
| 5 | 거의 매번 나옴 (ARC/retain cycle, struct vs class, Optional, @State vs @Binding, GCD/async-await) |
| 3 | 절반 정도 나옴 (Copy-on-Write, actor 격리, ViewBuilder, URLSession 구조) |
| 1 | 드물게 나옴 (String 그래핌 클러스터 내부, `_read`/`_modify` 접근자, 커스텀 Layout 프로토콜 세부) |

### 2.2 centrality — 개념 중심성 (가중치 25%)

"이걸 모르면 다른 개념도 무너지는가 / 다른 개념과 얼마나 연결되는가"

| 점수 | 기준 |
|---|---|
| 5 | 여러 주제의 뿌리 (값/참조 타입 구분, ARC, 프로토콜, SwiftUI 뷰 아이덴티티, Sendable) |
| 3 | 한 영역 안에서 중요 (PropertyWrapper 동작, TaskGroup, Codable 커스터마이징) |
| 1 | 고립된 지식 (특정 modifier의 기본값, 특정 API의 시그니처 암기) |

### 2.3 practicality — 실무 영향도 (가중치 20%)

"모르면 실무에서 버그·크래시·성능 저하로 이어지는가"

| 점수 | 기준 |
|---|---|
| 5 | 모르면 크래시/메모리 누수/데이터 레이스 (retain cycle, unowned 오용, MainActor 위반, @StateObject 오용으로 상태 초기화) |
| 3 | 성능 저하나 미묘한 UI 버그 (불필요한 뷰 재계산, CoW 미이해로 인한 복사 비용, List 아이덴티티 오류) |
| 1 | 알면 좋은 배경 지식 (컴파일러 최적화 내부, 역사적 배경) |

### 2.4 discrimination — 변별력 (가중치 15%)

"이 질문의 답변 수준이 초심자/숙련자를 가르는가"

| 점수 | 기준 |
|---|---|
| 5 | 이해 깊이가 그대로 드러남 ("왜 @State는 struct인 View에서 값을 유지할 수 있나", "actor reentrancy가 왜 문제인가") |
| 3 | 정확히 아는 사람과 대충 아는 사람이 갈림 (weak vs unowned 선택 기준) |
| 1 | 외우면 누구나 답함 (용어 정의) |

### 2.5 우선순위 점수와 티어

```
priority = 0.40·frequency + 0.25·centrality + 0.20·practicality + 0.15·discrimination
```

| 티어 | 점수 | 의미 |
|---|---|---|
| **S 필수** | ≥ 4.2 | 반드시 완벽하게. 항상 최상단 |
| **A 중요** | ≥ 3.4 | 우선 학습 대상 |
| **B 기본** | ≥ 2.6 | 기본기 확인용 |
| **C 지엽** | < 2.6 | 시간 남을 때. 항상 하단 |

- 섹션 화면에서 문제는 **priority 내림차순**으로 정렬된다 (앱이 자동 계산).
- ⚠️ 가중치를 기본값(40/25/20/15)에서 바꾸면 `app.js`의 `EVAL_WEIGHTS`와 `validate.py`의 `WEIGHTS`도 동일하게 수정할 것.

## 4. 생성/검수 체크리스트

새 문제를 추가하기 전에 전부 체크:

- [ ] 1문제 1개념인가? (두 개념이면 분리)
- [ ] 같은 개념의 기존 문제와 중복되지 않는가?
- [ ] 스템만 읽고 무엇을 묻는지 명확한가?
- [ ] 오답 3개가 모두 '흔한 오해' 기반으로 그럴듯한가?
- [ ] 정답이 길이·문체로 티 나지 않는가? 극단 표현 단서가 없는가?
- [ ] 해설이 모범답안 수준이고, 주요 오답의 이유를 포함하는가?
- [ ] `eval` 4개 축을 루브릭 기준표에 맞게 채점했는가? (점수 인플레이션 금지)
- [ ] `coverage`가 섹션 배분(50/30/20)을 깨뜨리지 않는가?
- [ ] `keywords`가 glossary에 실제 존재하는가? 새 용어는 glossary에 등록했는가?
- [ ] `correctIndex`가 한쪽으로 몰리지 않는가?
- [ ] `id`가 유일한가?
- [ ] 버전 기준(Swift 5.9+/6, iOS 17+ SwiftUI)에 맞는가? deprecated API를 현행처럼 출제하지 않았는가?

기계 검증(ID 중복, keywords/related 참조 무결성, correctIndex 범위·분포, 정답 길이 티, 부정형 비율)은 `python3 validate.py`로 실행한다.
